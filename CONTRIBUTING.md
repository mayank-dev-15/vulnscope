## 📜 Code of Conduct

Be respectful, constructive, and inclusive. Harassment or toxic behavior will not be tolerated.

## 💡 How Can I Help?

Here are some ways you can contribute:

- 🐛 **Report bugs** — Found something broken? Open an issue!
- ✨ **Suggest features** — Have an idea? Let's discuss it!
- 📖 **Improve documentation** — Typos, clarifications, examples all help
- 🔧 **Submit code fixes** — Fix bugs or add features via PRs
- 🌐 **Translate** — Help make the project accessible in more languages
- ⭐ **Star the repo** — If you like it, a star goes a long way!

## 🐛 Reporting Bugs

Before creating a bug report, please:

1. **Search existing issues** — Someone may have already reported it
2. **Check the README** — Your question might already be answered

When opening a bug issue, include:

- **Clear title** — Summarize the problem in one line
- **Description** — What happened vs. what you expected
- **Steps to reproduce** — Exact steps so I can replicate it
- **Environment** — OS, browser, version, etc.
- **Screenshots** — If applicable, add them

```
Example:
Title: [BUG] Login button unresponsive on Firefox 120

Description: Clicking the login button does nothing on Firefox 120.

Steps:
1. Go to /login
2. Enter credentials
3. Click "Login"
4. Nothing happens

Environment: Firefox 120, Ubuntu 22.04
```

## ✨ Suggesting Features

Feature requests are welcome! Please provide:

- **Clear title** — What you want
- **Use case** — Why this would be useful
- **Implementation ideas** — If you have any (optional)

## 🔧 Pull Request Process

1. **Fork** the repository
2. **Clone** your fork: `git clone https://github.com/YOUR_USERNAME/REPO_NAME.git`
3. **Create a branch**: `git checkout -b feature/your-feature-name`
4. **Make your changes** — Keep them focused and atomic
5. **Test** — Make sure everything works
6. **Commit** — Use clear commit messages:
   - `feat: add dark mode toggle`
   - `fix: resolve login timeout issue`
   - `docs: update installation guide`
7. **Push**: `git push origin feature/your-feature-name`
8. **Open a Pull Request** — Describe what you changed and why

### PR Checklist

- [ ] Code follows the project's style
- [ ] Tests pass (if applicable)
- [ ] Documentation updated (if needed)
- [ ] No merge conflicts
- [ ] PR description is clear and detailed

## 📐 Coding Standards

- **Commit messages:** Use semantic prefixes (`feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`)
- **Code style:** Follow the existing patterns in the project
- **Comments:** Explain "why", not "what"
- **Naming:** Use descriptive variable/function names
- **Keep it simple** — Readability over cleverness
